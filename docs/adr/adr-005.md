# ADR-005: Bring your own Agent vs Prompting Agent ourselves


## Status
Accepted.

## Context
The AI Agent needs some way to interact with our tool. 
There are 2 ways we can achieve this:
- We make a command to prompt the agent into tackling a task on the issue tracker. It completes the task in this active "runner" that loops and executes the AI. Basically, baton will be the actual program running the AI. Like Baton grabs the issue, boots up the Anthropic SDK for example, hands the issue to the LLM, and then logs the result.
- We don't run the AI agent ourselves. Instead there is a bring-your-own-agent approach. The user will use an external AI agent which the user would then give permissions to the agent to run the baton commands, and tell it something like "Update your progress on a task using this issue tracker whenever you make changes".

## Decision
We will have a bring-your-own-agent approach, where the end user will tell their AI Agent to use our CLI tool.

## Consequences

We decided to go for the bring your own agent approach because we believe this will lower our scope and needed effort. There is a lot more overhead and effort that goes into creating a SDK that will actually trigger the agent for our end user. It's much easier for the AI Agent to learn how to interact with our tool. This means that instead of building functions to call an Anthropic SDK for example, we instead make markdown files and --json flags to help the AI Agent use our tool effectively.
