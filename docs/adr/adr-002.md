# ADR-002: Use a Terminal CLI as the Primary Platform

## Status
Accepted

## Context
The team evaluated how the user would interact with the issue tracker. A VSCode extension would provide a rich UI within a popular editor, while a Terminal CLI would be simpler to build and ship. Our decision was guided by our limited timeline of 1 month, paired with a limited skill set where none of us have much experience developing for either platform. We also made considerations about how similar either platform would be to other issue trackers, e.g. Beads issue tracker.

## Decision
We will deliver the platform as a Terminal CLI rather than a VSCode Extension.

The terminal interface represents the lowest-complexity path to a working product and is the most feasible option given the time remaining. It avoids the overhead of VSCode extension APIs, packaging, and editor lifecycle management.

## Consequences

### Easier
- Faster to build and iterate, not forced to create a GUI
- Works across editors without being tied to VSCode
- More support / flexibility for building out the tool

### More Difficult
- Less intuitive for users who prefer GUI-based tooling
- More difficult to be creative / different than other existing tools
- Requires additional work to upgrade to a richer interface later, i.e. implementing a GUI
